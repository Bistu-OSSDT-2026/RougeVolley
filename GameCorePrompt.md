一、项目架构概览与设计原则
本项目基于 FXGL 游戏引擎构建一个程序化生成的地牢探索游戏，整体架构遵循事件驱动、组件化、分层解耦的设计哲学。代码生成 Agent 必须严格遵守以下核心原则，确保产出代码具备高内聚、低耦合、可测试与高性能特性。

核心架构特征
固定时间步长主循环：逻辑更新频率锁定为 60Hz，采用累加器模式处理帧率波动，保障物理模拟与游戏判定的确定性。
事件总线通信机制：模块间通过 EventBus 发布/订阅消息交互，禁止跨层直接调用，实现 UI、逻辑、渲染三层完全解耦。
组件化实体系统（ECS）：所有游戏对象（玩家、敌人、物品）均以 Entity 为基础，通过组合不同 Component 实现行为扩展，提升复用性与灵活性。
多线程任务调度：耗时操作（如 A* 寻路、BSP 分割）必须提交至后台线程执行，主线程仅负责调度与状态同步，避免卡顿。
全局设计约束
设计维度	强制要求
更新机制	必须使用固定时间步长（1/60 秒），结合时间累加器处理 delta time
并发模型	耗时计算移出主线程，使用异步任务或线程池管理
渲染表现	渲染层需支持插值算法，分离逻辑帧与渲染帧，保证视觉流畅
实体标识	所有 Entity 实例必须分配全局唯一 UUID，用于网络同步与状态追踪
可重现性	地图生成必须支持种子（Seed）控制，相同输入产生一致输出
内存管理	高频创建/销毁对象（如 BSPNode）应使用对象池技术减少 GC 压力
模块协作范式
系统由八大功能模块协同工作，形成完整闭环：

主循环驱动整个游戏流程；
场景管理维护当前活动场景栈；
实体工厂统一创建各类游戏实体；
敌人物品生成依据权重与空间规则填充内容；
BSP 地牢布局递归分割空间并生成房间走廊；
难度区域划分根据玩家位置动态调整挑战等级；
算法集成层提供性能优化与扩展接口；
Code Plan指导开发节奏与质量保障。
所有模块实现必须面向接口编程，暴露清晰契约，并通过独立单元测试验证核心逻辑正确性。

二、模块实现规范与编码指导
本节详细定义各核心模块的实现规范，为代码生成 Agent 提供精确、可执行的编码指令。所有实现必须严格遵循接口契约、数据结构定义与错误处理策略。

1. 主循环（Main Loop）
负责游戏整体调度，确保逻辑确定性与性能稳定。

更新机制：
逻辑更新频率固定为 60Hz（每帧约 16.67ms）。
使用时间累加器模式处理 delta time，伪代码如下： ```java accumulator += deltaTime; while (accumulator >= 1.0 / 60.0) { updateLogic(); accumulator -= 1.0 / 60.0; }
CODE
复制
- **事件驱动**：
    - 所有跨模块通信通过 `FXGL EventBus` 发布/订阅完成。
    - 禁止在主循环中直接调用 UI 或渲染层方法。
- **多线程处理**：
    - A* 寻路、BSP 分割等耗时操作提交至后台线程。
    - 主线程仅接收异步任务结果并同步状态。

### 2. 场景管理（Scene Management）

基于栈模型管理场景生命周期，支持无缝切换。

| 方法 | 触发时机 | 必须执行的操作 |
| --- | --- | --- |
| `onInit()` | 场景首次创建 | 加载静态资源（纹理、音效）、初始化配置参数 |
| `onEnter()` | 成为当前活动场景 | 恢复玩家状态、启动背景音乐、激活输入监听 |
| `onUpdate()` | 每帧调用 | 处理游戏逻辑、AI 行为、碰撞检测 |
| `onExit()` | 切换至其他场景 | 保存进度、释放临时资源、暂停音频 |


- 场景栈支持 `push(scene)` 和 `pop()` 操作，最大深度建议不超过 5 层。

### 3. 实体工厂（Entity Factory）

统一创建游戏实体，强制执行标准化流程。

- **接口定义**：
    ```java
    public interface EntityFactory {
      Entity createEntity(Type type, Point2D position);
    }
实现要求：
所有实体必须分配全局唯一 UUID。
实例化前校验 position 非空，否则抛出 IllegalArgumentException。
根据类型自动装配组件：
PLAYER：附加 PlayerControl, CameraFollow
ENEMY_*：附加 Collidable, AIControl, PhysicsComponent
ITEM_*：附加 ViewComponent, Pickup
4. 敌人物品生成（Enemy & Item Spawning）
依据权重与空间约束智能填充内容。

生成规则：
使用加权随机算法决定生成类型。
位置必须满足预设约束条件：
类型	权重	位置约束
普通敌人	70%	BSP 叶节点房间内部
精英敌人	20%	相邻房间连接点附近
Boss 敌人	5%	最大叶节点房间中心
普通物品	60%	走廊或房间边缘区域
稀有物品	30%	与精英敌人同区域
传说物品	10%	仅由 Boss 敌人掉落
敌人物品生成权重分布
类别	生成权重
普通敌人	70%
精英敌人	20%
Boss敌人	5%
普通物品	60%
稀有物品	30%
传说物品	10%
数据说明:

当前数据包含两类独立权重体系：敌人类型生成权重与物品稀有度生成权重，不构成单一整体分布。
敌人子类总计：95%（普通+精英+Boss），剩余5%可能为未定义或其他类型。
物品子类总计：100%（普通+稀有+传说），分布符合标准比例。
趋势：普通类（敌人/物品）均占主导地位，高稀有度单位（如Boss、传说物品）生成概率较低，体现逐级递减设计逻辑。
5. BSP 地牢布局（BSP Dungeon Generation）
递归分割空间，生成连通地牢结构。

核心步骤：
初始化根节点覆盖全图。
递归分割空间，优先切割长边。
在叶节点内生成房间（尺寸为节点的 60%-90%）。
连接兄弟节点房间中心，采用 L 形走廊。
后处理优化路径，清理死胡同。
关键数据结构： java class Rect { double x, y, w, h; } class BSPNode { Rect rect; BSPNode left, right; Room room; // 叶节点专属 }
参数范围：
递归深度：1–6（默认 4）
最小房间尺寸：20–80 像素（默认 40）
6. 难度区域划分（Difficulty Zones）
根据玩家距起点距离动态调整挑战等级。

区域	距离范围	特性描述
安全区	0–20 单位	无主动敌人，明亮环境，用于新手引导
普通区	20–40 单位	以普通敌人为主，基础装备掉落
危险区	40–60 单位	精英敌人占比 30%，稀有物品，复杂迷宫
极限区	60+ 单位	Boss 专属区域，传说物品，黑暗陷阱
7. 算法集成与扩展机制
支持性能优化与未来功能拓展。

性能优化：
使用迭代替代递归防止栈溢出。
对 BSPNode 实例应用对象池技术。
支持种子控制（Seed-Based Determinism）保证结果可重现。
采用增量生成策略，分块异步加载提升响应速度。
扩展能力：
允许热替换地牢生成算法（如迷宫、Voronoi 图）。
支持语义标注（如“宝库”、“监狱”）绑定装饰模板。
可接入 Perlin 噪声进行地形起伏与材质过渡处理。
8. 错误处理与健壮性要求
所有模块必须具备防御性编程意识。

参数校验：
所有公共接口必须验证输入参数有效性。
示例：createEntity(type, pos) 中若 pos == null，立即抛出 IllegalArgumentException。
边界条件处理：
BSP 分割时检查最小尺寸阈值，避免无效分割。
场景栈 pop() 前检查是否为空，防止异常中断。
异常传播策略：
可恢复错误记录日志并继续执行。
不可恢复错误（如资源加载失败）应向上抛出，由上层统一处理。
测试覆盖要求：
每个模块需提供独立单元测试。
关键路径包括：BSP 分割正确性、地图连通性（A* 验证）、1000 次生成平均耗时统计、极端参数边界测试。
